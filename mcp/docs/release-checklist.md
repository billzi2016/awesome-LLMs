# Release Checklist

Use this checklist before treating the repository as a publish-ready Awesome project.

## Content

- [ ] The repository scope is clearly stated and still matches the actual content.
- [ ] English and Chinese README files remain aligned in structure and meaning.
- [ ] New sections do not drift outside the MCP ecosystem boundary.
- [ ] Resource descriptions are factual, concise, and non-promotional.

## Links

- [ ] Core official links open successfully.
- [ ] Newly added GitHub repositories still exist and match their descriptions.
- [ ] Archived links are clearly marked as archived or historical.
- [ ] Internal links between README and `docs/` resolve correctly.

## Rendering

- [ ] Mermaid diagrams render correctly on the target platform.
- [ ] README table of contents anchors jump to the intended sections.
- [ ] Markdown tables render correctly on both GitHub web and mobile.
- [ ] Badge links and language-switch links work as expected.

## Curation Quality

- [ ] Official resources appear before secondary or community summaries.
- [ ] Security-sensitive resources do not point to mirrors or unclear download pages.
- [ ] Production-oriented resources are distinguished from archived examples when needed.
- [ ] Duplicated links are intentional and serve different sections.

## Maintenance

- [ ] Contribution guide and curation policy still match the current repository structure.
- [ ] Primer documents stay consistent with the main README.
- [ ] New protocol concepts or extensions are reflected in both language versions when relevant.
- [ ] The registry, SDK, and extension references are still current.

## Optional Final Pass

- [ ] Run a dedicated link checker if one is introduced later.
- [ ] Preview the repository on GitHub before publishing widely.
- [ ] Do a final manual skim for heading consistency, anchor naming, and terminology drift.
