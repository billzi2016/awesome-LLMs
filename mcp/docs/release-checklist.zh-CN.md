# 发布检查清单

在把仓库视为可发布的 Awesome 项目之前，先用这份清单做一次检查。

## 内容

- [ ] 仓库范围定义清晰，并且仍与实际内容一致。
- [ ] 英文和中文 README 在结构和含义上保持一致。
- [ ] 新增章节没有偏离 MCP 生态主题边界。
- [ ] 资源描述保持客观、简洁，不带宣传性表达。

## 链接

- [ ] 核心官方链接可以正常打开。
- [ ] 新增的 GitHub 仓库仍然存在，且与描述相符。
- [ ] 已归档链接被明确标注为归档或历史参考。
- [ ] README 与 `docs/` 之间的内部链接可以正确跳转。

## 渲染

- [ ] Mermaid 图在目标平台上可以正常渲染。
- [ ] README 目录锚点能跳到正确章节。
- [ ] Markdown 表格在 GitHub Web 和移动端都能正常显示。
- [ ] badge 链接和中英文切换链接工作正常。

## 收录质量

- [ ] 官方资源优先于二级总结或社区转述资源。
- [ ] 涉及安全敏感内容的资源没有指向镜像站或来源不清的下载页。
- [ ] 面向生产的资源与归档示例在需要时有明确区分。
- [ ] 重复链接如果存在，是出于不同章节语境的合理需要。

## 维护一致性

- [ ] 贡献指南与收录规则仍与当前仓库结构一致。
- [ ] primer 文档与主 README 保持一致。
- [ ] 新增的协议概念或扩展，在需要时同步反映到中英文版本。
- [ ] Registry、SDK、Extensions 的引用仍然是当前版本。

## 可选最终检查

- [ ] 如果后续引入了自动化工具，运行一次专门的链接检查。
- [ ] 发布前在 GitHub 页面上实际预览一次仓库效果。
- [ ] 最后再人工通读一遍标题一致性、锚点命名和术语是否漂移。
