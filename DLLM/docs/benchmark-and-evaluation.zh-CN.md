# Diffusion LLM 的 Benchmark 与评测

本文档讨论 Diffusion LLM 应该怎样评测，以及为什么拿它和自回归 LLM 做“简单分数对比”往往会误导。

最核心的一点是：

\[
\text{DLLM quality}
\neq
\text{单一指标}.
\]

评测时至少应该把这些维度分开看：

- 文本质量；
- 多样性；
- 可控性；
- 推理成本；
- 解码预算；
- 任务准确性；
- 在适用情况下的 likelihood 相关指标。

## 1. 为什么评测比看起来更难

Diffusion LLM 和自回归模型的差别不只是架构不同，还包括：

- 解码过程不同；
- refinement 预算不同；
- 并行程度不同；
- 是否允许后续修订不同；
- 训练目标和推理方式之间的关系不同。

因此，任何忽略推理预算的评测都不完整。

高层可以写成：

\[
\text{model quality}
=
f(\text{training}, \text{decoding}, \text{budget}, \text{task}).
\]

这也是为什么不能只看一个样例，或者只看一个 benchmark 分数，就下结论说某个 DLLM 更强。

## 2. 核心评测维度

### 2.1 文本质量

这一维通常包括：

- 流畅性；
- 语法正确性；
- 连贯性；
- 在适用情况下的事实一致性；
- 与 prompt 或条件输入的相关性。

对开放式生成，这一维往往会结合：

- 人工评测；
- 模型打分；
- 在有 reference 的任务上使用任务特定的 overlap 指标。

### 2.2 多样性

很多 diffusion 风格文本生成方法都会宣称：非左到右生成更容易得到更高的输出多样性。

但真正有用的问题是：

- 多次采样结果是否真的有意义地不同？
- 多样性是不是以连贯性下降为代价？
- 模型到底是“真正多样”，还是“只是更不稳定”？

这一维在下面这些任务中特别重要：

- 对话生成；
- 改写；
- 创意生成；
- 存在多个合理目标的条件生成任务。

### 2.3 可控性

有些 DLLM，尤其像 `Diffusion-LM` 这样的早期方法，主打的是 controllable generation。

这时评测要问：

- 输出是否满足目标属性？
- 控制是否破坏了流畅性？
- 更强的控制会不会损害语义忠实性？

如果论文主打的是“可控文本生成”，那这一维常常比一般 LM 指标更关键。

### 2.4 推理成本

对 DLLM 来说，这一维不是次要问题，而是中心问题。

关键成本变量包括：

- 去噪步数；
- 墙钟延迟；
- 前向调用次数；
- 显存占用；
- 长文本下的实际吞吐。

例如：

\[
\text{更高质量}
\text{ 可能只是来自 }
\text{更多 refinement steps},
\]

因此如果拿一个 32-step DLLM 去和一个 1-pass AR decode 比质量，而不讨论计算成本，通常是没有意义的。

### 2.5 Likelihood 相关指标

在适用时，也可以评测：

- negative log-likelihood；
- perplexity；
- calibration 一类指标。

这一维对那些显式优化 likelihood 或 ELBO 风格目标的方法尤其重要；对纯粹强调下游生成质量的论文，相对没那么核心。

## 3. 如何和 AR LLM 做公平比较

这是最容易出问题的地方。

### 3.1 Token 时间和 Refinement 时间不是一回事

自回归模型把时间花在 token 位置顺序上。

DLLM 把时间花在 refinement steps 上。

所以：

\[
\text{AR complexity is token-serial},
\qquad
\text{DLLM complexity is refinement-serial}.
\]

一个公平的比较，至少要明确说明：

- 输出长度；
- 去噪步数；
- 每一步是否处理所有位置；
- 是否使用 remasking；
- 实际运行时间。

### 3.2 参数量相等并不够

两个模型参数量差不多，也仍然可能无法直接比较，如果：

- 一个需要更多 refinement steps；
- 一个在去噪时拿到了双向上下文；
- 一个做了更强的后训练；
- 一个在评测时使用了更宽松的解码预算。

因此：

\[
\text{equal parameter count}
\neq
\text{equal inference condition}.
\]

### 3.3 步数相等也不够

即便去噪步数一致，也还是不一定公平，因为：

- 有的方法每一步更新所有 token；
- 有的方法每一步只更新 masked 位置；
- 有的方法使用自适应 remasking；
- 有的方法使用分块或位置偏置解码。

更接近公平的比较单位应该是：

- 总模型调用次数；
- 总 token-position update 次数；
- 墙钟延迟；
- 在条件允许时做硬件归一化吞吐比较。

## 4. 开放式语言建模评测

对一般语言建模而言，有用的评测维度包括：

- perplexity 或 likelihood proxy；
- 开放式生成质量；
- 长文本连贯性；
- 重复和退化行为；
- 在不同 decoding budget 下的样本效率。

在这个场景下，一篇 DLLM 论文最好回答四个问题：

1. 最终样本质量怎么样？
2. 需要多少 refinement steps？
3. 当预算缩小时，质量如何变化？
4. 和强 AR baseline 相比结果如何？

其中第 3 个问题经常被忽略，但非常重要。

## 5. 条件生成任务评测

对 seq2seq 风格 DLLM，评测会强依赖任务本身。

### 5.1 摘要生成

常见维度：

- faithfulness；
- coverage；
- 压缩质量；
- 事实一致性；
- 在使用时的 ROUGE 一类 overlap 指标。

### 5.2 机器翻译

常见维度：

- adequacy；
- fluency；
- BLEU 或相关 overlap 指标；
- 不同句长下的鲁棒性。

### 5.3 对话与响应生成

常见维度：

- 相关性；
- 多样性；
- 一致性；
- specificity；
- 在适用时的安全性。

### 5.4 改写与重写任务

常见维度：

- 语义保持；
- 词汇多样性；
- 风格或结构变化质量。

这些任务恰恰是 diffusion decoding 可能展现优势的地方，因为合理目标往往不止一个。

## 6. 大规模 DLLM 评测

像 `LLaDA` 这样的现代 DLLM，会让评测越来越接近大模型评测。

### 6.1 Instruction Following

需要问的问题包括：

- 模型是否遵守格式约束？
- 是否真的完成了用户要求的任务？
- 在长 prompt 或多轮场景下是否稳定？

### 6.2 推理能力

推理评测可能包括：

- 数学 benchmark；
- 多步推理任务；
- 对 chain-of-thought 敏感的任务；
- 现代 LLM 论文常用的 benchmark suite。

对 DLLM 来说，尤其要问：在有限去噪预算下，推理质量是否还能保持。

### 6.3 代码生成

代码任务特别考验：

- 语法正确性；
- 功能正确性；
- 长程依赖处理能力；
- iterative refinement 到底是在帮忙，还是在破坏精确性。

在原理上，diffusion 风格的“可修订性”可能是优势，但前提是解码本身足够稳定。

## 7. 多样性和质量之间的权衡

Diffusion 风格文本生成里一个常见说法是：它可能比严格左到右 decoding 更支持多样性。

这个说法不能直接接受，必须进一步问：

\[
\text{模型是真的更多样，还是只是更不稳定？}
\]

好的评测应把这些量区分开：

- 输出多样性；
- 语义正确性；
- 流畅性；
- 不确定性的校准情况。

否则，随机噪声式输出很容易被误读成“多样性更强”。

## 8. Step-Budget 曲线很重要

对 DLLM 来说，只报一个点是不够的。更强的评测方式，是报告质量如何随 inference budget 变化。

概念上：

\[
\text{quality} = g(K),
\]

其中 \(K\) 是去噪步数。

更有信息量的报告方式包括：

- 4-step 质量；
- 8-step 质量；
- 16-step 质量；
- 32-step 质量；
- 超过某个预算后的边际收益变化。

这能看出一个方法到底是：

- 天生高效；
- 只有在高计算量下才表现好；
- 在紧预算下是否依然稳健。

## 9. 人工评测仍然重要

自动指标当然有价值，但 DLLM 的错误分布常常和 AR 模型不一样。

人工评测在这些方面尤其重要：

- 长文本连贯性；
- 编辑质量；
- controllability 是否真的成功；
- 细微失败模式；
- revision 是否真的把样本改好了。

最佳实践不是用人工评测替代自动指标，而是把两者结合起来。

## 10. 一篇强 DLLM 论文的评测部分应该包含什么

一篇评测做得比较扎实的论文，理想情况下应该报告：

1. 用的是什么解码策略；
2. 去噪步数是多少；
3. 是否使用 remasking 或自适应日程；
4. 需要什么计算预算或延迟；
5. 当预算更小时，质量如何变化；
6. 用了什么 AR baseline，以及 baseline 的推理条件是什么；
7. 是否同时覆盖质量和多样性；
8. 是否包含人工评测。

如果这些信息缺失很多，那么论文里的比较结论就要谨慎看待。

## 11. 最小总结

最短但有用的一句话是：

\[
\text{评估 DLLM，本质上是在同时评估模型本身和它的解码预算。}
\]

这就是它和“直接看 benchmark 分数”之间最大的差别。
