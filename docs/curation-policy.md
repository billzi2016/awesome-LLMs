# Curation Policy

This document defines how resources are included in this project.

## Inclusion Criteria

A resource may be included when it satisfies most of the following:

- directly relevant to the repository's stated scope;
- for skill-focused themes, exposes a reusable skill concept, structure, or implementation pattern;
- has a stable and accessible source;
- provides a paper, codebase, documentation, benchmark, dataset, registry entry, template, or technical explanation;
- is useful for research, engineering, evaluation, or learning;
- for workflow-oriented themes, is useful for authoring, packaging, operating, discovering, or evaluating the target workflow;
- has an objective description that can be verified from the source;
- does not introduce copyright, license, privacy, or platform-policy risk.

## Priority Resources

Higher priority is given to:

- foundational papers;
- official implementations;
- official documentation, specifications, or built-in platform systems;
- maintained libraries;
- reproducible experiments;
- clear templates, manifests, packaging conventions, or registries when they are core to the topic;
- widely used tutorials or notes;
- benchmark and evaluation resources;
- applications that clearly belong to the repository topic.

## Rejection Criteria

A resource should not be included when it is:

- only weakly related to the repository scope;
- for skill-focused themes, a generic prompt collection without reusable structure;
- for skill-focused themes, a broad AI tools list with no clear skill-specific value;
- inaccessible or unverifiable;
- a duplicate of an existing item;
- mainly promotional;
- missing technical detail;
- a copied PDF, mirrored dataset, unofficial model-weight upload, or protected marketplace content;
- likely to violate copyright, licenses, privacy, or platform rules.

## Link Policy

Prefer original sources in this order:

1. arXiv, DOI, publisher, official paper page, or official platform/provider documentation
2. official author, organization, or code repository
3. official documentation, registry, marketplace, or catalog page
4. official dataset, benchmark, or packaging/runtime-integration page
5. maintained project page
6. secondary summary page

If a link cannot be verified, do not add it.

## Description Style

Descriptions should be:

- factual;
- concise;
- neutral;
- specific about the method, task, contribution, runtime boundary, or authoring value when applicable.

Avoid unsupported claims such as "best", "must-read", "revolutionary", "state-of-the-art", or "the future of agents" unless the wording is tied to a specific source-backed statement.

## Scope Notes

This repository family includes both research-heavy themes and workflow-heavy themes.

Examples that often belong:

- papers, benchmarks, datasets, and official implementations tied to the topic;
- official documentation, specifications, or platform-native systems;
- templates, registries, and operational guides when they are core to the theme;
- concrete examples that demonstrate the target workflow or capability clearly.

Examples that often do not belong:

- generic background material with no clear topic fit;
- vague resource lists with no reusable structure;
- unverifiable mirrors or reposted protected content.

## Maintenance

The list should be reviewed periodically for:

- broken links;
- duplicated entries;
- outdated descriptions;
- archived repositories;
- moved paper or project pages;
- inconsistent English and Chinese content.
